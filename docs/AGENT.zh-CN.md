## 1. 这是什么软件

**Subnetra** 是一个用**纯 Zig**（锁定 2026 最新标准库 `std.posix`）编写的虚拟三层（Layer 3）
自适应组网工具。它采用**星型拓扑（Hub-and-Spoke）**，在物理专线之上构建虚拟子网，通过一条
私有的、全加密的 UDP 隧道转发裸 IP 包。

它面向**通用 Linux 环境**，包括资源极度受限的容器（BusyBox / RouterOS Container）。最终
产物是**一个静态链接的单二进制文件**，零第三方依赖。

权威设计文档是 [`subnetra-develop.md`](subnetra-develop.md)（PRD & 架构）。该文档是“要构建
**什么**”的唯一事实来源；本文件约束你在构建过程中“**必须如何行动**”。

## 3. 不可妥协的铁律

以下条款直接来自 PRD，具有**强制约束力**。无论结果看起来多么“干净”，违反其中任何一条都是硬性失败。

1. **零第三方依赖。** 不依赖 WireGuard，不引入 `ikcp.c`，不使用网络框架，不引入外部加密库。
   只使用 Zig 标准库与经 `std.posix` 的原生系统调用。v2 可靠层也必须自研（arena 版 ARQ），
   绝不 vendoring 任何 C 库。
2. **分层零动态内存分配。** **数据面**（`reactor`、`crypto`）严格零分配：所有数据包缓冲区
   在启动时锁死于常驻内存。控制面/可靠层路径可使用独立 arena，但绝不得污染数据面的内存线。
3. **单线程、无锁反应堆。** 单线程；一个无锁、零分配的就绪循环。单线程、无锁、每包零分配
   这一不变量**才是**铁律；具体的就绪原语在 **`comptime` 按平台选择**：Linux 用 epoll
   边缘触发（`EPOLLET`），macOS 用 `poll(2)`（`kqueue` 推迟到后续里程碑）。
   **禁用线程，禁用锁。** 策略热更新通过原子指针交换（RCU）完成，绝不使用互斥量。
4. **无状态混淆 / 隐蔽。** ChaCha20-Poly1305 全加密，密文中**无任何魔数**。认证失败时
   **静默丢弃**——绝不回发 TCP Reset、ICMP 或任何可被观测的响应。
5. **传输安全在 v1 即为强制项。** PSK 密钥、每端 64-bit **绝不复用的单调递增 nonce**，以及
   滑动窗口**防重放**校验。这些不得推迟。
6. **单个极简二进制。** 默认 `-O ReleaseSmall`，零第三方依赖（铁律 #1 在所有平台依旧成立）。
   **Linux：** 基于 musl-libc 完全静态链接，`ldd` 显示 `not a dynamic executable`，
   目标 **≤ 512KB**。**macOS：** 最小动态链接——**仅**链接 `libSystem`（Apple 不提供静态
   libc），并记录其自身的体积基线。`ldd` 静态检查是 **仅限 Linux** 的门禁。
7. **坚持测试驱动。** 遵循 PRD 中的 TDD 流程。纯逻辑必须随附测试。任何提交前，
   `zig build test` 必须保持全绿。
8. **无状态、无握手传输。** 每个数据报自描述、可独立解码——每包携带的 `epoch`
   **本身就是**完整的会话建立机制。Subnetra **不做任何建链往返、不做挑战/应答、
   不做带内会话协商**——v1 不做，v2 也不做。可靠层状态（未来的 ARQ/FEC）可置于
   独立 arena，但绝不得演变为握手；任何未来传输模式均由**静态的 per-link 配置**
   选择（复用预留的 `negotiation_version`/`flags` 字段），绝不在线缆上协商。
   两项后果**作为设计取舍被接受，而非推迟**：（a）在途攻击者可重放一个尚未被对端
   观测到的 epoch 的抓获报文，从而**瞬时**迁移对端端点——它会在对端下一个真实
   报文到达时自愈，且 off-path 攻击者无法伪造（issue #42）；（b）某节点墙钟在重启间
   回退时会被对端拒绝，直至其时钟前进越过旧值——由 NTP/RTC 在运维层面缓解
   （见 `docs/deployment.md`），绝不靠协议内的 epoch 交换。

## 4. 范围：v1 与 v2——不得越界

- **v1（必须交付）：** `raw_direct` 数据面 + PSK 加密 + 防重放 + RCU 热更新策略引擎。
- **v2（Roadmap，仅留接口）：** `kcp_arq`、`fec_xor`——自研可靠层模式，
  **由静态 per-link 配置选择，绝不走线缆握手**（见铁律 #8）。在 v1 中**只预留**
  `egress` 分支与报头 `negotiation_version`/`flags` 字段，**不实现**它们。
  v2 分支返回 `error.NotImplemented`。**Roadmap 上没有握手**；握手本可带来的
  对 issue #42 与时钟回退的对称修复，被有意**不予实现**（铁律 #8）。

不得提前动工 v2，不得把 v2 的复杂度塞进 v1，不得引入握手，也不得删除 v2 的预留点。

- **只做精准、对齐目标的改动。** 不做无关重构，不做“顺手重排全世界”的提交，不做投机性抽象。
- **遵守语言策略。** 所有代码、注释与根级文档使用**英文**。中文文件仅有两份：
  `docs/subnetra-develop.md`（设计文档）与 `docs/AGENT.zh-CN.md`（本镜像）。不要翻译设计文档；
  但要保持本镜像同步。
- **进度如实。** `README.md` 中的开发进度表反映真实进展。任务落地时如实更新——绝不把占位
  标记为完成。
- **先验证再宣称完成。** 在声明某任务完成前，先构建、跑 `zig build test`，并确认二进制仍为
  静态链接且体积在预算内。

## 6. 发布与版本规范

发布版本号只存在于**唯一一处**：[`build.zig.zon`](build.zig.zon) 的 `.version`
字段。它在构建期通过 `build_options` 模块注入守护进程横幅——**绝不要在 `src/`
中硬编码版本字符串。**

发布前：

1. 将 `build.zig.zon` 的 `.version` 升到新的 `X.Y.Z`（语义化版本）。
2. 通过正常 PR 流程把该改动提交到 `main`。
3. 给该 commit 打标签 `vX.Y.Z`——标签**必须**等于 `v` + `build.zig.zon` 中的版本。
   发布工作流有一个守卫作业，二者不一致时会让发布失败，因此错配的标签永远不会发布。
4. 推送 `v*` 标签会触发 `.github/workflows/release.yml`，构建四架构静态二进制、
   GHCR 多架构镜像，以及可 `docker load` 的离线分架构镜像包，并连同合并的
   `SHA256SUMS.txt` 一起发布到 GitHub Release。

未先把 `build.zig.zon` 升到对应版本，不得创建 `v*` 标签。
